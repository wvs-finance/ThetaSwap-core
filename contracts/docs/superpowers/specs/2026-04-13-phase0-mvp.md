## 1. Purpose

Prove end-to-end that V_B's economic mechanism works: a deposit of asset1 appreciates over time at a rate proportional to a globalGrowth signal, and a redemption returns the deposit plus the realized appreciation. Everything else in Rev 10.3 is safety and UX scaffolding around this core flow; nothing else matters until this works.

Phase 0 produces one passing integration test against a known-good reference implementation. If the test passes, the economic model is validated and the peripheral layers (timelock, entry window, admin rotation, observability, Panoptic integration, RAN product) can be specified and implemented against a known-working foundation.

---

## 2. Scope

### In scope

- **V_B_MVP contract:** ESR subclass that restricts `gulp()` to an immutable `scaler` address. No other logic.
- **Scaler_MVP contract:** reads a mock `globalGrowth` value each period, computes `donation = notional × ΔglobalGrowth × scale` per Rev 10.3 §9.2 math, pulls asset1 from a funding source, transfers, calls `gulp()`.
- **Mock EVC** (~5 LOC): satisfies ESR's `EVCUtil` constructor `_evc != address(0)` check. Never called on hot paths (verified against actual source).
- **Mock Consumer:** returns a scripted `globalGrowth` curve for tests.
- **Mock FundingSource:** holds asset1, approves Scaler_MVP to pull.
- **Python reference implementation:** mirrors the Scaler formula and ESR smear math. Produces the expected asset1 amount at redeem time given inputs.
- **Forge integration test** that deploys the stack, runs the golden path, and asserts the on-chain result matches the Python reference within ±1 bps.

### Out of scope for Phase 0 (deferred to later phases per Rev 10.3)

- Withdrawal timelock (`requestUnlock` / 14d wait / finalize) — test direct `redeem` on ESR
- Entry-window gating on `deposit` / `mint`
- Admin and scaler rotation (`proposeScaler`, `cancelScalerProposal`, `rotateScaler`, `initScaler`)
- `IERC4626TimelockVault` interface
- `IRateProvider` adapter
- Observability (`lastGulpAt`, `gulpStaleness()`, `pokeStaleness()`, `GulpStale` events)
- Seed-share lock to `0xdead` — plain seed without lock for MVP
- ERC-20-only asset1 validation
- Panoptic CT integration
- RAN product contract and real funding source
- WithdrawalTimelockMod

### Deferred numeric commitments

Phase 0 uses synthetic test values for `V_B_liquidity_notional`, `productScale`, and oracle conversion. Calibration of production values remains the Rev 10.3 §9.2 econometric workstream.

## 3. Success Criterion

**One Forge test passes with a ≤1 bps discrepancy against the Python reference.**

Concrete assertion:

```
|redeemedAssets_onchain - redeemedAssets_reference| / redeemedAssets_reference ≤ 1e-4
```

The test deploys the full stack, exercises the lifecycle below, reads the on-chain result, compares to the Python reference computed over the same inputs. No other success criteria. No other acceptance requirements.

## 4. Components

### 4.1 V_B_MVP.sol

```
contract V_B_MVP is EulerSavingsRate {
    address public immutable scaler;

    constructor(
        IEVC _evc,
        address _asset,
        string memory _name,
        string memory _symbol,
        address _scaler
    ) EulerSavingsRate(_evc, _asset, _name, _symbol) {
        require(_scaler != address(0), "ZeroScaler");
        scaler = _scaler;
    }

    function gulp() public override {
        require(msg.sender == scaler, "NotScaler");
        super.gulp();
    }
}
```

Size: ~25 LOC including imports and comment. Only behavior change vs. plain ESR is the scaler gate on `gulp()`.

Notes:
- Does not apply its own `nonReentrant` on `gulp()` — ESR's inherited guard is sufficient. Nested `nonReentrant` would revert (verified per installed source).
- No `withdraw`/`redeem` override — ESR's native behavior is the tested path.
- No entry-window check on `deposit`/`mint` — ESR native.

### 4.2 Scaler_MVP.sol

```
contract Scaler_MVP {
    address public immutable vault;
    address public immutable consumer;
    address public immutable fundingSource;
    address public immutable asset;
    uint256 public immutable liquidityNotional;
    uint256 public immutable productScale;
    uint256 public immutable epochDuration;

    uint256 public lastEpochClose;
    uint256 public lastGrowthSnapshot;

    function advanceEpoch() external {
        require(block.timestamp >= lastEpochClose + epochDuration, "Early");

        uint256 growthEnd = IConsumer(consumer).globalGrowth();
        require(growthEnd >= lastGrowthSnapshot, "NotMonotone");

        uint256 deltaGrowth = growthEnd - lastGrowthSnapshot;
        uint256 rewardAsset0 = FullMath.mulDiv(liquidityNotional, deltaGrowth, 1 << 128);
        // For Phase 0: assume asset0 == asset1 (parity pools). No oracle conversion.
        uint256 donation = FullMath.mulDiv(rewardAsset0, productScale, 1e18);

        IERC20(asset).transferFrom(fundingSource, vault, donation);
        IV_B_MVP(vault).gulp();

        lastEpochClose = block.timestamp;
        lastGrowthSnapshot = growthEnd;
    }
}
```

Size: ~50 LOC with imports, events, errors.

Phase 0 simplification: asset0 == asset1 (parity assumption — run the test with a pair where this holds, e.g., the vault's underlying equals what globalGrowth is denominated in). The asset0→asset1 oracle is deferred to Phase 1.

### 4.7 Forge integration test (contracts/test/phase0/V_B_MVP.golden-path.t.sol)

```
function test_golden_path() public {
    // Deploy stack
    MockEVC evc = new MockEVC();
    MockERC20 asset = new MockERC20(...);
    MockConsumer consumer = new MockConsumer();
    MockFundingSource funding = new MockFundingSource(asset, 1_000_000e18);

    Scaler_MVP scaler = new Scaler_MVP(consumer, funding, asset, NOTIONAL, SCALE, 14 days);
    V_B_MVP vault = new V_B_MVP(evc, address(asset), "V_B MVP", "vbMVP", address(scaler));
    // Wire scaler vault pointer via constructor or post-deploy setter

    // Seed
    asset.mint(admin, 1e7 + 1);
    asset.approve(address(vault), type(uint256).max);
    vault.deposit(1e7 + 1, admin);

    // User deposits
    uint256 userDeposit = 100e18;
    asset.mint(user, userDeposit);
    vm.prank(user);
    asset.approve(address(vault), userDeposit);
    vm.prank(user);
    uint256 shares = vault.deposit(userDeposit, user);

    // Advance time and globalGrowth
    for (uint i = 0; i < 3; i++) {
        vm.warp(block.timestamp + 14 days);
        consumer.set(consumer.globalGrowth() + GROWTH_PER_EPOCH_Q128);
        scaler.advanceEpoch();
    }

    // Wait past final smear
    vm.warp(block.timestamp + 14 days + 1);

    // Redeem
    vm.prank(user);
    uint256 assetsReceived = vault.redeem(shares, user, user);

    // Assert against Python reference
    uint256 expected = _loadReferenceResult();  // from JSON fixture generated by reference.py
    assertApproxEqRel(assetsReceived, expected, 1e14);  // 1 bps = 1e-4 = 1e14 out of 1e18
}
```

## 5. The golden-path lifecycle

Simplified timeline the test exercises:

```
t = 0
  Deploy stack, seed vault with 1e7+1 assets (admin).

t = 1 hour (arbitrary)
  User deposits 100 asset1, receives 100 shares (1:1 at virgin-ish vault).
  Vault state: totalAssets = 1e7+1 + 100, totalSupply = 1e7+1 + 100.

t = 14 days
  Scaler reads globalGrowth = G1. Computes donation = f(notional, G1, scale).
  Pulls asset1 from funding source, calls vault.gulp().
  ESR: interestLeft = donation, smearEnd = t + 14 days.

t = 14 days + 7 days
  Half of donation has streamed into totalAssets. Share price has risen.

t = 28 days
  Scaler reads globalGrowth = G2. Computes donation_2 = f(notional, G2-G1, scale).
  Layered smear: interestLeft = unsmeared_remainder + donation_2, smearEnd = t + 14 days.

...

t = N×14 days + 14 days + 1 second
  Final smear fully drained. totalAssets = initial deposits + all donations.
  User redeems shares, receives assets.

Assertion: assetsReceived matches Python reference within 1 bps.
```

- The economic mechanism is proven against a reference implementation
- `V_B_liquidity_notional` and `productScale` parameters are concretely wired and testable
- The Scaler ↔ V_B integration surface is exercised
- The Python reference becomes the test oracle for all subsequent phases
- Peripheral layer specs (timelock, entry window, admin rotation, observability, Panoptic) can be written with known-working fundamentals underneath

If Phase 0 fails, no peripheral layer work is justified until the failure is understood.

## 9. Open decisions for Phase 0

- **Parity assumption.** Scaler_MVP assumes asset0 == asset1 for the globalGrowth → asset1 conversion. Pick a test pair where this holds, or add a trivial 1:1 oracle mock. Defer the real oracle to Phase 1.
- **`liquidityNotional` and `productScale` test values.** Use stubs calibrated to produce observable appreciation over a handful of epochs (e.g., ~1–5% per epoch). Real calibration remains Rev 10.3 §9.2 econometric work.
- **Growth curve.** The test scripts a synthetic curve. The curve is a test input, not a product parameter.
- **Python version and tooling.** Use the existing project Python convention (frozen dataclasses, full typing — per the functional-python skill).

---

## 10. What this spec is not

This is not a BTT spec. It does not enumerate all edge cases, all revert paths, all fuzz scenarios. It specifies the minimum viable integration test that proves the core economic mechanism. BTT specs for V_B (with all Rev 10.3 features) come after Phase 0 passes.
