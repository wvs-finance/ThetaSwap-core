### CR-P5 (MAJOR) — Step 0 ordering ambiguity in Task 11.N.2b
- **Location:** Draft §4 Task 11.N.2b Step 0 (line 134) vs Step 2 (line 136).
- **Issue:** The draft says Step 0 (schema migration) "is committed separately… downstream Steps 1-7 blocked on Step 0 green". Step 2 is the Dune-credit-budget probe and may HALT. Step 1 is the basket-address verification + design memo and may HALT. If Step 0 commits the new tables BEFORE Step 1 / Step 2 verify the data is actually fetchable, the canonical DuckDB ends up with two empty tables and a relaxed CHECK admitting two new `proxy_kind` values that have no rows. This is "partially-applied state" and TW already flags it in the decomposition trade-off block (line 161) but does not resolve it. The right ordering is likely: Step 1 (basket-address resolution + design memo, no schema change) → Step 2 (credit-budget probe, no schema change) → Step 0 (now safe to commit, all upstream HALT-points cleared) → Steps 3-7. Renaming "Step 0" is awkward (the name comes from Rev-5.2.1 Task 11.N.1's pattern), so consider a Step labeling like "Step 1 (pre-commit)", "Step 2 (probe)", "Step 3 (schema)", "Step 4 (test)" etc. — or, equivalently, gate the Step 0 commit on Step 1 + Step 2 PASS.
- **Fix:** Add to Step 0: "**Step 0 commit gate:** Step 1 (basket-address verification PASS) and Step 2 (Dune-credit-budget probe PASS, sufficient budget) MUST both green before Step 0 commits. The Rev-5.2.1 'Step 0 first' ordering held under a single-RPC-source backfill where the schema unblocked downstream code; under Task 11.N.2b's two-source data fetch + non-trivial credit budget, Step 0 ordering inverts to 'Step 0 commits last among the schema-migration trio'." This preserves atomicity.

### CR-P6 (MINOR) — Resume-mechanism additive-append not strict enough
- **Location:** Draft §3 Task 11.N.1b "Files" line 54 + "Non-negotiable rules" line 76.
- **Issue:** Files block says "preserve existing 52,068 rows byte-exact (additive-only — any mutation of an existing row aborts)". Non-neg rules block says "existing 52,068 rows in `onchain_copm_transfers` preserved byte-exact". Both are good. But Step 1 mocks the checkpoint at block 30,486,127 and asserts "the first RPC call uses block 30,486,128" — this is a test that the resume orchestrator reads the checkpoint correctly. There is NO test that confirms after resume completion the existing CSV is byte-exact equal to the pre-resume CSV on rows 1..52068. A Data Engineer could construct a fetcher that re-pulls overlapping ranges (e.g., a Covalent paginator that starts on block 30,486,000 — 128 blocks earlier than checkpoint — and silently dedupes by hash, mutating row order). The dedup logic is the place where additive-only could quietly break.
- **Fix:** Add a Step 1 sub-assertion: "After `resume_copm_transfers_backfill()` completes, `pd.read_csv(copm_transfers.csv).head(52068)` is byte-exact equal to the pre-resume snapshot at commit `f1f114cd1`." This forces the implementer to either pull strictly above `last_completed_end_block` or to dedup-and-discard rather than dedup-and-merge.
