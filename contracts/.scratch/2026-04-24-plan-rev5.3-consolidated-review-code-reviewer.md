## 3. New findings introduced by the brainstorm-fold

### CR-CF-1 (MAJOR) — `proxy_kind` admitted-set deviates from canonical design doc

**Location:** Draft L196 vs design doc §2 (L56) + §4 (L86-89).

**Issue:** Draft enumerates **eleven** admitted `proxy_kind` values (2 pre-existing + 9 new). The 9 new include `carbon_basket_user_volume_usd_copm_only` as a separate value distinct from `carbon_basket_user_volume_usd`. Design doc does NOT define `_copm_only` as a separate `proxy_kind` — design doc §2 lists exactly 8 new values (basket-aggregate + arb + 6 per-currency) and §4 says the COPM-pass-pass branch writes to `carbon_basket_user_volume_usd` *filtered to COPM-only* (same proxy_kind, different filter), not a new proxy_kind. The draft introduces a 9th value not in the canonical immutable design.

**Why it matters:** Design doc §3 says modification of pre-committed thresholds requires a "new design-doc revision + 3-way review cycle" — the same anti-fishing discipline applies to the proxy_kind set. A plan-side schema deviation from the canonical design that has not been routed through design-doc revision violates the brainstorm-fold's explicit immutability claim ("Brainstorm-converged design at `…/2026-04-24-carbon-basket-xd-design.md` (immutable)"; status-line L51).

**Fix options:** (a) Drop `carbon_basket_user_volume_usd_copm_only` from the CHECK admitted-set and let the COPM-only filter live as a row-attribute (e.g., a `filter_tag` column or composite primary key extension) under the same `proxy_kind = "carbon_basket_user_volume_usd"`; (b) Author a design-doc Rev-1.1 amendment adding the COPM-only proxy_kind explicitly, route through 3-way review, then justify in this plan. Option (a) preserves the immutable-design claim; option (b) requires extra review cycles.

### CR-CF-2 (MINOR) — Stale supply-channel-parallel language in pre-fold sections

**Location:** Draft §0 changelog L26 (PM-P3 paragraph) + §2 bullet (7) L62.

**Issue:** Both paragraphs still describe the OLD (pre-brainstorm-fold) policy: "11.O remains on supply-channel surrogate until both 11.N.1b and 11.N.2b.2 land". The brainstorm-fold delta at §0.1 (L42) + new §6 (L369-391) + 11.N.2c DAG block (L331) all explicitly RETIRE this policy. The user prompt for this review explicitly listed this as a stale-references audit concern.

**Why it matters:** A reader scanning §0 changelog or §2 bullet 7 sees the old policy stated in the present tense without any "SUPERSEDED by §0.1 / §6" annotation. The §0 fix-pass content is "preserved byte-exact" per the self-audit (L419), which is fine for audit-trail purposes — but absent a SUPERSEDED marker, it reads as a contradiction.

**Fix:** Add a single-sentence annotation to L26 and L62: "(SUPERSEDED by brainstorm-fold §0.1 + §6 — supply-channel-parallelism RETIRED.)" Preserves the audit trail while preventing the contradiction.

### CR-CF-3 (MINOR) — Step-number cross-reference off-by-one in atomic-commit handoff

**Location:** Draft L199, L214, L216, L235 all refer to "Task 11.N.2b.2 **Step 6**" as the post-ingestion atomic-commit step. Actual 11.N.2b.2 task at L256-257: Step 5 = "atomic schema-migration commit to canonical DB"; Step 6 = "run + verify against canonical DB" (post-commit verification).

The off-by-one is harmless to outcomes but confuses the rollback narrative (the L260-264 Step Atomicity Protocol correctly says "Step 5 atomically commits BOTH schema migration AND populated tables" — which contradicts the four upstream cites).

**Fix:** Change L199, L214, L216, L235 from "Step 6 (post-ingestion)" to "Step 5 (atomic transaction wrapping schema migration + row population, post-Step-4 smoke-probe)".
