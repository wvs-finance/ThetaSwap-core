## 2. Findings

**PM-P1 (BLOCKER — Step-0 schema-migration atomicity, applies to 11.N.2b).** TW Task 11.N.2b Step 0 commits the third schema migration *separately* (admits four `proxy_kind` values: `net_primary_issuance_usd`, `b2b_to_b2c_net_flow_usd`, `carbon_basket_rebalancing_count`, `carbon_basket_rebalancing_volume`). If Step 0 commits, then Step 2 (Dune-credit probe) HALTs, the schema is left admitting two values that no row populates. This is a *latent* state, not an immediate bug, but it (a) inflates the DDL fingerprint without justification, (b) creates a confusing audit trail (next reviewer asks "why is this CHECK relaxed if no rows use it?"), and (c) breaks the Rev-5.2.1 precedent where Step-0 immediately precedes a working pipeline. **Fix:** require Step 0 to commit ONLY after Step 1 (basket-address verification + golden-fixture memo) succeeds — schema migration moves from "before any code" to "after pre-commitment memo, before any data pull." This preserves atomic-rollback (the memo is reversible at the `.scratch/` level) without leaving an empty schema relaxation in committed DDL.

**PM-P2 (FLAG — Y-target deferral creates DAG ambiguity).** The Rev-5.2.1 PM-P1 DAG-clarification said 11.O can run in parallel on the supply-channel surrogate while 11.N.1 lands. Rev-5.3 reaffirms this for 11.N.2b but the underlying claim is now weaker: the Carbon-basket reframing implies the Y_asset_leg formula is *operationally wrong* under the recommended X_d (Carbon-basket rebalancing → CELO-vol or Mento-peg-spread Y, per A1 in the amendment-rider queue). If 11.O begins authoring on (supply-channel X, USD-COP-carry Y) and the recommended (Carbon-basket X, crypto-vol Y) lands later, 11.O's amendment-rider absorbs *two* scope shifts simultaneously. The Rev-5.2.1 amendment-rider mechanism was sized for one shift (X_d swap), not two (X_d + Y swap). **Fix:** Rev-5.3 should explicitly downgrade the parallel-11.O claim to "11.O on the supply-channel X + Y_asset_leg is now SPECULATIVE and likely Rev-5.4-superseded; orchestrator MAY defer 11.O dispatch until 11.N.2b lands and Rev-5.4 (Y-target re-pick) commits, accepting the timeline cost." Without this downgrade, the plan promises a parallelism that the amendment-rider mechanism cannot deliver cleanly.

**PM-P3 (BLOCKER — amendment-rider queue is not formally tracked).** TW added a 7-item A1-A7 queue inside the revision-history bullet. By the time Rev-5.4 lands, that queue will be 1-3 revisions old and buried inside a dense bullet. Standard PM practice is to lift queues like this into a dedicated section with a stable anchor. **Fix:** create a top-level section `## Amendment-Rider Queue (Rev-5.3 deferred)` containing A1-A7, OR move them into a checklist file `contracts/.scratch/2026-04-24-rev5.4-amendment-rider-queue.md` that Rev-5.4 explicitly consumes. The current draft has them in §5 of the *insertion draft* — but the insertion-target plan has no such section. After insertion, A1-A7 will live only as a numbered list inside the revision-history bullet, which is exactly the failure mode flagged.

**PM-P4 (BLOCKER — 11.N.2b decomposition).** TW recommends single-task with implementer-discretion split. PM disagrees: this task carries **three** atomic-rollback concerns simultaneously (schema migration to 4 `proxy_kind` values; new Dune-credit-budget probe with HALT-on-shortfall; 6-step golden-fixture + ingestion + filter + aggregation pipeline). The Rev-5.2.1 Task 11.N.1 precedent for "single task with Step-0 separately committable" was sized for ONE atomic concern (RPC backfill against a known-row-count target). 11.N.2b adds two more (credit budget unknown; basket-address verification not yet complete). **Fix:** force the split into 11.N.2b.1 (Step 0 + Step 1 + Step 2 = schema migration + basket-address pre-commitment + credit-budget probe; commit at HALT-decision boundary) and 11.N.2b.2 (Steps 3-7 = ingestion + filter + aggregation, predicated on 11.N.2b.1 returning a clean credit-budget GO). Task count becomes **60**, which is a small price for atomic-rollback hygiene. The TW "implementer-discretion split" is a non-decision that pushes risk onto the implementer.

**PM-N1 (NIT — does Carbon X_d obviate 11.N.1b distribution-channel?)** Per report §4 Option 1, Carbon basket-rebalancing is the *recommended* primary; report §4 Option 3 (COPM-only Carbon trade count) is sensitivity-only. The COPM Transfer-channel X_d (which 11.N.1b finishes) is now neither primary nor sensitivity per the research report — it is a third candidate retained only because Rev-5.2.1 already pre-committed to producing it. Recommendation: keep 11.N.1b but note explicitly in its Rationale that the *resulting* distribution-channel X_d is third-rank under the post-11.N.2 hierarchy, not co-equal-secondary as Rev-5.2.1 promised. Avoid wasting Task 11.O resolution-matrix slots on a deprecated candidate.

**PM-N2 (NIT — resumability test for 11.N.1b checkpoint).** Step 1 mocks the checkpoint file with a fixture pointing at block 30,486,127. But what if the *real* checkpoint file is corrupted, missing, or has a different schema (e.g., older field names)? **Fix:** add to Step 1 explicit assertions: (a) corrupted JSON → orchestrator HALTs with `CHECKPOINT_CORRUPTED`; (b) missing file → HALT with `CHECKPOINT_MISSING_RESTART_FROM_TASK_11.N.1`; (c) schema-mismatch (unexpected field name) → HALT with explicit field-list dump. None of these should silently default to block 0.

**PM-N3 (NIT — recovery-protocol path-switching criterion).** Six paths in 11.N.1b but the criterion for switching from path N to path N+1 is implicit. Specifically: a partial Covalent success (80k of 110k rows) leaves the project in a hybrid state — does the orchestrator append the partial 80k and then probe path 3 (Flipside) for the remaining 30k? Or does it discard the partial 80k and restart on path 3 from the post-52k checkpoint? **Fix:** explicit decision: per-path success means full coverage of the post-checkpoint range to chain-tip; partial coverage on a path is a path-FAILURE that drops back to the prior checkpoint, no hybrid-stitch.

**PM-N4 (NIT — meta-test for cumulative test-suite green).** Test count by Rev-5.3 is well past 30+ across `scripts/tests/inequality/`, `scripts/tests/remittance/`, root. Currently no plan rule mandates `pytest scripts/tests/` runs all-green at every commit boundary; the silent-test-pass guards from Rev-3.1 are per-file. **Fix:** add to Non-Negotiable Rule 1 (STRICT TDD) a sub-clause: "every plan-task commit must include a worknotes line confirming `pytest contracts/scripts/tests/` exits 0." Cheap belt-and-suspenders against accumulating disabled tests.

**PM-N5 (NIT — bot-research artifact governance).** The Task 11.N.2 research report at `contracts/.scratch/2026-04-24-copm-bot-attribution-research.md` (157 lines, 19KB, uncommitted) is the load-bearing trigger evidence for two new tasks. Standard PM practice: trigger evidence committed to git so audit trail is intact. **Fix:** Rev-5.3 commit should include the research report alongside the plan changes, OR the plan's revision-history bullet should explicitly note the report's `.scratch/` path with a sha256 or wc-l stamp so post-hoc tampering is detectable. Recommend the former (commit it).

**PM-N6 (NIT — spec-coverage self-check stale).** Per prior PM-Rev-5 review, the spec-coverage self-check section currently says 46 tasks. Rev-5.3 doesn't update it; doc-debt now spans Rev-5 → Rev-5.3 (4 revisions). **Fix:** Rev-5.3 either updates the self-check OR explicitly adds it to the Rev-5.4 amendment-rider queue (becomes A8). Continuing to defer is now a process smell.

- 11.N.2b.1: Step 0 (schema migration) + Step 1 (basket-address verification + pre-commitment memo) + Step 2 (Dune-credit-budget probe). Atomic-commit at "GO/NO-GO decision on production pull." Task count unit.

TW's "implementer-discretion split" pushes structural risk onto the implementer, who is acting under credit-budget pressure and likely under-decomposes. Plan count: **60 active** (57 + 11.N.1b + 11.N.2b.1 + 11.N.2b.2). The +1 count cost is trivially worth the rollback hygiene.

## 5. Risk Register (top 3 execution risks for Rev-5.3 as drafted)

**R1 — Dune-credit budget shortfall (HIGH).** Report §0 records 5.9 / 15 credits used. 11.N.2b production pull of full Carbon `tokenstraded` + `arbitrageexecuted` event tables across 10-token basket is open-ended (basket-wide event count "materially larger" than COPM-only's 1.7M). Pre-emptive mitigation: PM-P4 split forces a credit probe BEFORE production pull commits the plan to the path. Without the split, Step 0 schema migration commits, Step 2 HALTs on credit, downstream remediation requires either user budget top-up or path-switch to Flipside (recovery-protocol mitigation #b) which is unverified. **Probability post-fix:** medium; severity: medium (manageable with 11.N.2b.1 as gating task).

**R2 — Y-target double-shift (HIGH).** Per PM-P2: Carbon X_d invalidates Y_asset_leg formula. 11.O dispatched in parallel on the supply-channel surrogate will produce a Rev-2 spec that is operationally wrong on TWO axes (X and Y) once 11.N.2b lands. Amendment-rider absorption of two simultaneous scope shifts is unprecedented in the Rev-5.x history. **Mitigation:** PM-P2 fix (defer 11.O until Rev-5.4 Y-target re-pick) trades timeline for spec correctness. **Probability post-fix:** low; severity: high (a wrong-Y Rev-2 spec triggers a full Rev-3 spec authoring cycle = Rule 13 cycle-cap exposure).

**R3 — Amendment-rider queue rot (MEDIUM).** PM-P3: A1-A7 in revision-history-bullet form with no top-level anchor. By Rev-5.5+, queue items lose visibility, get implemented partially, and the canonical "what's deferred" record fragments across multiple revisions. **Mitigation:** PM-P3 fix (top-level Amendment-Rider Queue section OR external `.scratch/` checklist). **Probability post-fix:** low; severity: medium (silent scope creep into Rev-5.4 if not formalized).
